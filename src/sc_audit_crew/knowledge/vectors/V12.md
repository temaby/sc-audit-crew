# V12 — Forced State Changes

## What to check
Identify any external actor who can push the contract into a different state WITHOUT the affected user's knowledge or consent.

1. Admin functions that forcibly alter user positions, balances, or entitlements — trace the full economic outcome for the affected user.
2. Anyone-callable functions that change protocol state affecting users (public functions resetting timers, modifying rates, or toggling flags affecting user rewards).
3. Token donation attacks — sending tokens directly to the contract address to manipulate accounting that relies on `balanceOf(address(this))`.
4. Flash loan + forced state push — single-transaction sequence that puts the protocol into an edge state exploiting forced changes.

For each path identified:
- Trace the full economic consequence: what do affected users gain or lose?
- Is the harm permanent or reversible?
- Does the user have any recourse?

## Severity guidance
- External actor can permanently destroy user funds with no user consent or recourse: High or Critical.
- External actor can temporarily lock user funds: Medium.
- Admin can force-exit users at market disadvantage — apply CODE DEFECT vs ADMIN MALICE gate in calibration.
- Token donation that breaks accounting in a material way: Medium if exploitable, Low if marginal.
