# V19 — Spec vs Implementation

## Pre-condition check
If `{specification}` is empty or fewer than 50 words:
record status="not_applicable", summary="No formal specification provided; cross-check not possible."
Do NOT attempt to infer a specification from the code itself.

## What to check
For every explicitly stated invariant, parameter, or rule in the specification:

1. Fees and rates — does the implementation use exactly the fee percentages and basis points stated?
2. Thresholds — do numeric limits (max amounts, min periods, rate caps, decimal counts) match the spec exactly?
3. Accounting rules — does the calculation algorithm match the described method step-by-step?
4. Role permissions — do access control modifiers and role assignments match the documented role boundaries?
5. Edge case handling — does the spec describe behaviour for edge inputs (zero amounts, maximum values, empty records) that the code handles differently?

Flag any divergence as a logic finding with:
- The specific spec text (quoted or paraphrased) that is contradicted.
- The specific code behaviour that differs.
- The economic consequence of the divergence.

## Severity guidance
- Fee implementation extracting more than spec states, benefiting protocol at user expense: High.
- Critical accounting rule implemented differently (could cause insolvency or fund loss): High.
- Numeric threshold off from spec by a small margin: Medium.
- Undocumented parameter differing from implied spec behaviour: Medium or Low.
- Documentation error where spec is wrong and code is correct: Info.
