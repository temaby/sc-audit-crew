# V08 — DoS

## What to check
1. Unbounded loops over storage arrays — if an array grows without bound, iteration will eventually hit the block gas limit.
   - Special rule: if the iteration is inside a view function, the calibration task applies the UNBOUNDED ARRAY IN VIEW FUNCTION minimum severity (Medium). Flag it clearly.
2. Push payments to external addresses — sending ETH/tokens to a contract address that may revert (griefing). Single failed recipient blocks all recipients in a loop.
3. Forced revert griefing — can an attacker put the contract into a state where a critical function always reverts?
4. Mapping enumeration anti-pattern — any pattern requiring enumeration of all mapping keys is a DoS vector; mappings are not iterable.
5. Block gas limit arithmetic — for bounded loops, estimate the maximum iteration cost and confirm it fits within typical block limits.

## Severity guidance
- Unbounded loop that permanently blocks all user withdrawals: High.
- Unbounded loop in view function disabling front-ends and monitoring: Medium (minimum per calibration rule).
- Single-recipient push payment that can be griefed with negligible systemic impact: Low.
- Bounded loop with documented size cap and verified gas budget: Info.
