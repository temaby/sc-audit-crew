# V11 — Admin Setter Validation

## MANDATORY CROSS-INVARIANT ENUMERATION

Work through each step for every setter in the contract:

### Step 1 — Identify invariant pairs

Find state variables that form logical pairs or groups:

- Lower bound + upper bound on the same quantity (e.g., minStake ≤ maxStake)
- Fee rates that must sum to ≤ 10000 basis points (or ≤ 100%)
- Cooldown period that must be shorter than a lock period
- Start timestamp must be < end timestamp
- Slippage tolerance that must be ≤ 100%

### Step 2 — Write each invariant in plain English

Example: "minStake must always be ≤ maxStake"

### Step 3 — Check setter validation

For each setter: does it validate the invariant against the CURRENT stored value of its counterpart before writing the new value?

### Step 4 — Emit if violated

If any setter allows a violating update without reverting: HIGH finding.
Rationale: an admin making two separate, individually-valid calls can put the contract into an inconsistent state (e.g., minStake > maxStake) where no user transaction can succeed, with no recovery path.

## TIME PARAMETER INVARIANTS (PT-F)

Add these to the cross-invariant enumeration for any function accepting schedule parameters:

1. Start timestamp backdating — can `startTimestamp` be set to a past value with no validation?
   - If yes: a newly created record carries an immediately claimable balance at creation time.
   - Severity: MEDIUM regardless of whether documentation permits backdating.
   - Note any explicit documentation permission as a mitigating factor in the description, but preserve the MEDIUM rating.

2. Zero-rate schedule (INPUT VALIDATION ROOT CAUSE) — does the creation function unconditionally accept any combination of interval and total-amount parameters, including combinations that produce a per-interval payout of zero for any token at any decimal count?
   - This is an input-validation defect: the contract should assert a minimum effective payout at record-creation time and revert if the configuration would distribute nothing.
   - Focus the finding description on the MISSING REVERT / MISSING GUARD. Do NOT describe this as a decimal-scaling or arithmetic-truncation issue — that framing belongs to V15 (PT-D). The two findings have different root causes and different fixes.
   - Severity: MEDIUM (not Low). Recommend a minimum-rate guard in the creation function.

## MISSING EVENTS

Critical parameter changes must emit events. Missing event on a privileged setter: Low.
Parameters requiring events: fee rates, access control transfers, token addresses, rate limits.

## Zero-value validation

For every setter parameter: is zero a valid or invalid state for this variable?
If zero is invalid (e.g., fee divisor, token address): the setter must revert on zero input.
Silently accepted zero that corrupts protocol state: Medium or High depending on impact.

## Severity guidance

- Setter allows invariant violation that permanently blocks user transactions: High.
- Zero-value accepted for a parameter where zero is invalid and causes state corruption: Medium or High.
- Backdated start timestamp with no validation: Medium.
- Zero-rate schedule creatable without revert: Medium.
- Missing event on non-critical setter: Low.
