# V15 — Decimal Scaling

## HARDCODED MAGNITUDE CONSTANT CHECK

Locate every hardcoded constant (1eN, 10\*\*N, PRECISION, DENOMINATOR, SCALE) used to scale token amounts.

For each constant, apply this decision tree (no external calls needed):

1. Does the contract call `token.decimals()` and derive the scale dynamically from that return value?
   YES → safe, no finding.

2. Does the contract have an explicit decimal guard (e.g., `require(token.decimals() == 18)` in constructor/initializer)?
   YES → safe, no finding.

3. Neither: the contract assumes fixed decimals with no runtime protection.
   Determine severity:
   a. Token address is hardcoded AND its decimals are visible in the same source file → compare directly.
   Confirmed mismatch = Critical.
   b. Token injected via constructor/parameter OR decimals not visible in source →
   emit Medium: "Contract hardcodes a 1eN scaling constant but accepts an arbitrary token without validating its decimal count. Any token with different decimals causes systematic mis-scaling of all amounts."

Also flag thresholds defined in raw units while the rest of the contract uses scaled units (or vice versa).

## RATE FLOOR CHECK (PT-D)

Applies to: protocols with lockup, vesting, or streaming creation functions.
If no such functions exist in the audited contract: record this sub-check as not applicable and move on.

For every lockup, vesting, or streaming creation function:

Step 1 — Identify the expression computing tokens per interval, e.g.:
perInterval = totalAmount \* intervalSecs / durationSecs

Step 2 — Apply the arithmetic test with a realistic low-decimal scenario:
totalAmount = 1 token at 6 decimals = 1,000,000 (1e6)
durationSecs = 10 years = 315,360,000 seconds
intervalSecs = 1 second
perInterval = 1,000,000 / 315,360,000 = 0 → ZERO RATE

Step 3 — If the result is zero for any realistic low-decimal, small-amount combination:
emit MEDIUM regardless of whether a decimal mismatch was also found.
Focus the finding description on the ARITHMETIC TRUNCATION ROOT CAUSE: the formula `totalAmount * intervalSecs / durationSecs` evaluates to zero due to integer division when units are small relative to the duration. Do NOT describe this as a missing input-validation guard — that framing belongs to V11 (PT-F). The two findings have different root causes and different fixes.

## Severity guidance

- Confirmed decimal mismatch (hardcoded scale vs actual token decimals): Critical.
- Hardcoded scale constant with no decimal validation, arbitrary token injection: Medium.
- Zero-rate schedule creatable without revert: Medium.
- Threshold defined in mixed units (raw vs scaled): Medium.
