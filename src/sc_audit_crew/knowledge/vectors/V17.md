# V17 — Initializer Security

Applies ONLY to upgradeable proxy contracts.
If source code contains no `initialize()`, no `upgradeTo`/`upgradeToAndCall`, and no
Proxy / TransparentUpgradeableProxy / UUPSUpgradeable / BeaconProxy base contract:
record status="not_applicable", summary="No proxy pattern detected."

## What to check
1. Single-use enforcement — does the `initializer` modifier (from OpenZeppelin Initializable) prevent re-initialization?
   Specifically: can `initialize()` be called a second time on the proxy?

2. `_disableInitializers()` in implementation constructor — without this call, anyone can:
   a. Call `initialize()` directly on the implementation contract.
   b. Use the initialized implementation for a malicious `delegatecall` that corrupts the proxy's storage.
   This is High severity — the implementation should be permanently locked at deploy time.

3. Re-init vectors after upgrade — can a new implementation version reinitialize already-initialized storage,
   overwriting user balances, token addresses, or access control? Look for `reinitializer(N)` modifiers
   and verify the version check is robust.

4. Constructor vs initializer confusion — state set in the implementation constructor does NOT appear in
   the proxy's storage. Any initialization logic must be in the `initialize()` function, not the constructor.

## Severity guidance
- Implementation initializable by anyone, enabling storage manipulation via delegatecall: High or Critical.
- `_disableInitializers()` missing from implementation constructor: High.
- Reinitializable storage that could overwrite user balances: High.
- Constructor used for initialization state in an upgradeable contract: Medium.
- Minor initializer ordering concern with no direct exploit: Low.
