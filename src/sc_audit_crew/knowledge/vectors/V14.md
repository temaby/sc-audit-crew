# V14 — Non-Standard ERC20

## FOUR MANDATORY SUB-CHECKS (PT-E)
Record each sub-check attempt in your V14 coverage entry summary.
For any sub-check not applicable, state explicitly why (e.g., "no deposit function exists").

---

### [ ] FEE-ON-TRANSFER
For every function that transfers tokens INTO the contract AND adds the amount to an internal accounting variable:

Pattern to detect:
```
internalVar += amount;
token.transferFrom(sender, address(this), amount);
```
With a fee-on-transfer token: actual received = amount − fee.
internalVar accumulates more than the contract holds. Over time, the last withdrawer fails (balance < internalVar).

Emit Medium if this pattern is found and the contract documentation does not explicitly restrict to non-fee-on-transfer tokens.

---

### [ ] REBASING TOKEN
For contracts that maintain an internal accounting variable representing all outstanding entitlements:
- Positive rebase (balance increases externally): excess tokens are stuck — no internal entitlement covers them.
- Negative rebase (balance decreases externally): contract becomes insolvent — last claimants cannot withdraw.

Detection: if the contract sums all outstanding entitlements into an internal counter and gates withdrawals on it, a rebasing token silently breaks the solvency invariant with no on-chain error.

Emit Medium if no documentation restriction against rebasing tokens AND no constructor/setter validation of this property.

---

### [ ] TWO-ADDRESS PROXY BYPASS
Some tokens expose two addresses sharing the same underlying balance (proxy + implementation).
Any withdrawal function guarded solely by an address-inequality check against the primary token can be bypassed by supplying the alternate address.
Correct guard: pre/post balance comparison on the primary token address, not address equality.

Flag as Medium if applicable and not already restricted to admin-only callers.
If admin-only: apply the MANDATORY DOWNGRADE RULE in the calibration task.

---

### [ ] ERC777 CALLBACK REENTRANCY
For every admin withdrawal and every user withdrawal function:

FIRST: verify CEI. If all guard state is committed before the transfer → defense-in-depth only, no finding.

For admin withdrawal functions where surplus is NOT committed before transfer:
Apply the CODE DEFECT vs ADMIN MALICE gate (in calibration):
- A well-intentioned admin calling the function triggers the ERC777 callback automatically — no malice required. This is a CODE DEFECT. Do NOT downgrade. Emit HIGH.
- If triggering the re-entry requires deliberate admin abuse beyond documented usage: apply MANDATORY DOWNGRADE RULE → Medium.

For user withdrawal functions where state guarding double-withdrawal is written AFTER the transfer: emit High.

---

## GENERAL NOTES
- BOUNDARY: V14 covers ERC20 transfer()/transferFrom() return value checks. Low-level call() return values belong to V02.
- Tokens that return false instead of reverting (USDT, BNB): SafeERC20 or explicit return-value check is required.

## Severity guidance
- Fee-on-transfer causing ledger insolvency accumulating over time: Medium.
- Rebasing token breaking solvency invariant: Medium.
- Two-address bypass (non-admin accessible): Medium.
- ERC777 callback exploitable by arbitrary user: High.
- ERC777 callback triggered by non-malicious admin (CODE DEFECT): High.
