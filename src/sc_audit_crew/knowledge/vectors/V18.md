# V18 — State-Lock DoS

## What to check
Identify boolean flags or status enums that can be set to a terminal value with NO recovery path:

1. Protocol pause flags — can the contract be permanently paused? Is there an unpause function accessible to any trusted party?
2. Individual account lock flags — can user accounts be permanently locked with no dispute or appeal mechanism?
3. Status enums with terminal states — can a record advance to "Cancelled", "Revoked", or "Expired" from which withdrawals are permanently blocked?
4. Admin-triggered permanent states — an admin calling a function that permanently blocks all user withdrawals with no recovery.

For each identified state-lock:
- Is there a recovery function? Who can call it?
- What is the economic consequence of permanent lock? Do users permanently lose access to funds?
- Apply the CODE DEFECT vs ADMIN MALICE gate in calibration: if a non-malicious admin can trigger this state through documented usage, it is a code defect, not an admin risk.

## Severity guidance
- Permanent protocol-wide lock causing all user fund loss with no recovery: High (CODE DEFECT gate applies if triggerable by documented admin usage).
- Permanent individual account lock with no dispute mechanism: Medium.
- Temporary lock with a documented and accessible recovery path: Low.
- Pause function with a timelock-gated unpause: Low or Info.
