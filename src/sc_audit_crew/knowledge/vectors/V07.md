# V07 — Integer Edge Cases

## CRITICAL PATTERN — Intermediate type truncation

> **COMMON MISCONCEPTION**: Solidity ≥0.8 does NOT prevent sub-256-bit overflow — it converts
> silent wrap-arounds into REVERTS (panic 0x11). A `uint112 × uint112` product that exceeds 2^112
> causes the transaction to revert unconditionally, permanently bricking any function that calls it.
> Do NOT conclude "safe" or "mitigated" because the compiler version is 0.8+.

In Solidity, arithmetic is performed at the OPERAND type's precision — it is NOT promoted to uint256.

### Procedure

For every multiplication `a * b` in the contract:

1. Identify the declared Solidity types of both `a` and `b`.
2. If either operand is NOT uint256 (e.g. uint8, uint32, uint64, uint96, uint112, uint128):
   a. Compute the maximum realistic product under normal protocol inputs.
   b. Compare against the type's maximum (uint128 max ≈ 3.4 × 10^38, uint112 max ≈ 5.2 × 10^33, etc.).
   c. If the product can exceed the type max under realistic inputs:
   - In Solidity ≥0.8 outside unchecked blocks: the multiplication REVERTS (panic), permanently bricking any function that calls it.
   - This is a REVERT, not a silent wrap-around. The SOL08-OVERFLOW gate in the calibration task must NOT reject these findings.
   - Default severity: HIGH.
   - Peer reviewer / calibration task may upgrade to Critical if the bricked function is a primary withdrawal or claim path and the revert is unconditional with no workaround.

### Fix guidance (prose only — do not include code)

Describe that the fix is to cast at least one operand to uint256 before the multiplication, performing the product in 256-bit space before dividing or assigning.

## Other integer checks

1. `unchecked {}` blocks — every unchecked block must have a comment explaining why overflow is mathematically impossible. Without it: Low.
2. Division before multiplication — when the ordering causes unintentional precision loss in a final result (not intentional rounding): Medium.
3. Multi-step precision loss — if truncation accumulates across multiple steps to a material token amount: Medium.

## Severity guidance

- Sub-256-bit type multiplication causing unconditional revert on withdrawal/claim: High (potential Critical upgrade if all users affected unconditionally).
- Unchecked block without safety comment: Low.
- Division-before-multiply causing unintentional precision loss: Medium.
- Precision loss with no material financial impact: Low or Info.
