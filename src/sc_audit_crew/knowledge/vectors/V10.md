# V10 — Business Logic

## MANDATORY FOR ALL PROTOCOL TYPES

### No asset-management logic escape hatch

If the contract handles no token balances and performs no fee calculations:
record status="not_applicable", summary="Contract contains no asset-management logic; fee and invariant checks not applicable."

### Fee calculation check

Verify no rounding error systematically favours the protocol over users.
A consistent 1-wei-per-operation bias compounds to a material amount at scale: High finding.

### Core invariant check

Identify the core accounting invariant for this protocol type and verify it holds after every state-mutating function:

- AMM: k = reserve0 × reserve1 must hold after every swap (allowing for fees)
- Lending: totalBorrowed ≤ totalDeposited at all times
- Vesting/streaming: sumEntitlements ≤ token balance at all times
- Staking: sum of all user stakes == reported totalStaked

### Rounding direction check

Division truncation must round in the direction that protects users:

- Lending: round UP for debt (user owes more), round DOWN for rewards (user receives less)
- AMM: round amounts-in UP and amounts-out DOWN to protect the pool invariant
- General principle: rounding must never systematically advantage the protocol at user expense

## VESTING-SPECIFIC EXTENSION

Apply ONLY if the protocol type provided in your task contains "vesting", "streaming", or "lockup".
Skip the sections below entirely for all other protocol types.

### Termination function check (PT-A)

Identify every function whose purpose is to terminate, revoke, cancel, slash, or remove a user entitlement — regardless of its name. Look for: revoke, cancel, terminate, forfeit, slash, withdraw-for-admin, remove, close, wipe, zero-out, early-exit, or any function that deletes or resets a user record.

For each such function, verify end-to-end storage writes:

1. Are tokens ALREADY VESTED (accrued, streamed) but NOT YET WITHDRAWN preserved for the user?
2. Is "earned but unclaimed" correctly separated from "future entitlement"? Only future entitlement should be removed.

HIGH finding: function zeros or deletes the full entitlement record without first crediting already-earned tokens — those tokens become permanently unrecoverable.

Correct behaviour: user retains access to earned tokens even after termination. Future entitlement is what gets cancelled.

**MANDATORY TRACE — do NOT conclude "sound" by inspection alone:**
Identify the exact expression decremented from the reserve/accounting counter (e.g. `numTokensReservedForVesting -= X`).
Determine what `X` equals at runtime:

- Correct: `X` = total allocation − already-vested (equals only the FUTURE unvested portion)
- Incorrect: `X` = total allocation − amount-already-withdrawn (includes earned-but-unclaimed tokens)
  Common error: using `finalVestedAmount` (total at schedule end) minus `amountWithdrawn` gives a larger `X` than the truly unvested portion, silently confiscating earned tokens from the user. This produces a HIGH finding even if the code looks superficially correct.

### Activation guard completeness (PT-B)

After verifying token preservation, check whether deactivation leaves the record key reusable:

1. Enumerate all fields that the record-CREATION function tests to prevent duplicate entries for the same key (address, ID, slot).
2. Every such guard field must be either: (a) fully reset to zero/initial value during deactivation, OR (b) an explicit admin function exists to reset it.
3. If any creation-guard field retains a non-zero value after deactivation with no reset path: emit Medium. The key is permanently blocked from re-use.

**MANDATORY FIELD AUDIT — do NOT conclude "reusable" without tracing each field:**

- Identify the exact modifier or `require` used to guard record creation (e.g. `hasNoClaim` checking `_claim.startTimestamp == 0`).
- List the value of EVERY field it checks immediately AFTER the deactivation function returns.
- If any checked field is non-zero and the creation function reverts on non-zero input for that field: emit Medium immediately. A common pattern: `isActive = false` is written but `startTimestamp` is left non-zero, permanently blocking re-creation for that key.

## Severity guidance

- Core accounting invariant broken by standard function call: High or Critical.
- Systematic rounding bias in fee calculation at scale: High.
- Vesting termination destroying earned-but-unclaimed tokens: High.
- Activation guard preventing key reuse with no workaround: Medium.
- Occasional minor invariant deviation with no compounding effect: Low.
